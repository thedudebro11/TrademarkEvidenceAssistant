# Design Language

This document defines the personality of Trademark Evidence Assistant.

It is not a UI style guide.

It defines how the software should feel.

---

# Primary Goal

The application should feel:

calm

trustworthy

methodical

focused

transparent

professional

helpful

Everything should reinforce confidence.

Nothing should create anxiety.

---

# Design Philosophy

The software should never compete for attention.

The evidence is the focus.

The interface quietly supports the evidence.

---

# Visual Hierarchy

Evidence first.

Everything else second.

The preview is always the largest element.

Metadata supports the preview.

Questions support the metadata.

Actions support the questions.

---

# One Decision At A Time

Never overwhelm the user.

One evidence item.

One review.

One decision.

Then continue.

---

# Plain Language

Avoid technical jargon whenever possible.

Prefer:

Customer Photo

instead of

Associated Marketing Asset

Prefer:

Possible Invoice

instead of

Commerce Artifact

Everything should read like natural English.

---

# Explain Everything

Never display unexplained scores.

Never display unexplained confidence.

Never display unexplained suggestions.

Every suggestion should answer:

Why?

---

# Transparency

The application never hides:

why something was suggested

why something scored highly

why confidence changed

where information came from

The software should always be explainable.

---

# Respect The Evidence

Original files are treated as historical records.

They are never modified.

The interface should communicate this often.

Trust comes from consistency.

---

# Calm Colors

Avoid highly saturated colors.

Prefer neutral backgrounds.

Use color sparingly.

Color communicates meaning.

Not decoration.

Suggested meanings:

Blue

Information

Green

Confirmed

Amber

Needs Review

Red

Potential Problem

Gray

Archived

---

# Motion

Animations should be subtle.

Fast.

Purposeful.

Never distracting.

Animation exists to reinforce state changes.

Never entertainment.

---

# Typography

Readable above all else.

Avoid decorative fonts.

Hierarchy should come from:

spacing

weight

alignment

not excessive sizing.

---

# Empty States

Empty states should guide.

Not apologize.

Example:

"No related evidence has been linked yet."

Better than:

"No data found."

---

# Error Messages

Errors should explain:

what happened

why

how to recover

Never expose technical stack traces to the user.

---

# User Trust

Trust is the application's most valuable feature.

Trust is earned by:

predictable behavior

clear explanations

visible filesystem safety

consistent terminology

honest uncertainty

The application should never pretend to know more than it actually does.

---

# Final Principle

Every screen should help the user answer one question:

"Does this evidence strengthen my trademark evidence package, and why?"

If a feature does not support answering that question, it probably does not belong in Version 1.
